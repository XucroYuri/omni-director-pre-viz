# 里程碑计划（Vibe Coding 友好）

## 0. 一句话目标

以最小风险和最短路径达成商业可交付能力，并形成可持续迭代节奏。

## 1. 非目标

- 不进行大规模架构重写。
- 不新增外部 SaaS。

## 2. 里程碑拆分

### M1（1-2 周）：P0 可交付

- 范围：P0-1~P0-4
- 退出条件：
  - 关键流程均有状态与恢复路径
  - alert/confirm 清零
  - 导出可验证
- 验收材料：
  - `docs/delivery-checklist.md`
  - 演示路径：脚本 → 拆解 → 母图 → 视频 → 导出

### M2（2-3 周）：P1 体验优化

- 范围：P1-5~P1-7
- 退出条件：
  - 空状态+引导覆盖全链路
  - 长耗时反馈可见
  - 文案/术语一致
- 验收材料：
  - `docs/ux-copy-states.md`
  - 体验走查（新用户无说明完成流程）

### M3（2-3 周）：P2/P3 稳定扩展

- 范围：P2-8~P3-11
- 退出条件：
  - 诊断导出与支持入口上线
  - 最小合规提示上线
  - 可访问性达标
- 验收材料：
  - 诊断导出示例
  - 可访问性检查表

## 3. 依赖与风险

- M1 依赖统一消息体系与任务状态一致化。
- M2 依赖 M1 的状态规范与文案一致性。
- M3 依赖基本流程稳定后再加运营与合规。

## 4. 里程碑责任人建议

- M1：核心研发 + 体验负责人
- M2：体验负责人 + PM
- M3：运维/合规 + 研发

## 5. 验收策略

- 每一里程碑均使用 `docs/delivery-checklist.md` 作为验收基准。
- 里程碑完成后需补充“风险与回滚”记录。
